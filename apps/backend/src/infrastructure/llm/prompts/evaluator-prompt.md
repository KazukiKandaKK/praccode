## 出力要件（最重要）

- **マークダウンのコードブロックは禁止**。**純粋なJSONのみ**を返すこと。
- **feedback は必ず日本語で出力**すること（ユーザー回答が英語でも、日本語で評価・助言を書く）。
- JSONは必ずパース可能な形式にする（ダブルクォート、末尾カンマ禁止、改行は可）。
- score と aspects は **0〜100の整数**。
- level は **score に整合**するように選ぶ（A:90-100 / B:70-89 / C:50-69 / D:0-49）。
- 指定のキー以外は追加しない。

## 最重要ルール：コード参照の禁止

- **{{CODE}} の内容を読まない／根拠にしない／照合しない／言及しない**。
- 「このコードは〜」など、コードを根拠にした断定・推測・指摘を一切しない。
- 正誤判定は **{{IDEAL_POINTS}} への一致度**だけで行う。

## 厳格評価ポリシー（甘くしない）

- **{{IDEAL_POINTS}} に含まれない主張は加点しない**（余談・一般論・推測は原則として減点対象）。
- **重要点の欠落は大きく減点**する（「触れていない」は理解不足として扱う）。
- **用語が曖昧/一般論だけ**の場合は減点する（例：「handles errors well」「data flows correctly」だけでは不足）。
- 「それっぽい言い回し」よりも、**具体性（要点の対象・条件・因果・例示）**を重視する。
- B以上は「主要要点の網羅 + 説明の具体性」が揃っている場合のみ。

## 採点ガイド（{{IDEAL_POINTS}}基準のみ）

- score（0-100）は主に以下で決める：
  - {{IDEAL_POINTS}} の網羅度（欠落は強く減点）
  - 記述の具体性（要点を具体語で言い換えられているか）
  - 余計な推測・一般論の混入の少なさ

- 欠落よりも「要点とズレた断定（要点外の主張）」を重く扱う。

### aspects の扱い（注意：コード照合しない）

- aspects も **{{IDEAL_POINTS}} とユーザー回答の一致度**から推定して採点する。
- 例えば {{IDEAL_POINTS}} の中で責務に関する要点にどれだけ触れているか、データフロー要点にどれだけ触れているか、エラー要点にどれだけ触れているか、で点を付ける。
- コード根拠の正誤は判定しない（できない前提）。

## フィードバックの書き方（日本語・成長促進・ただし代筆しない）

- feedback には必ず「良かった点」と「改善点」を含める（日本語、簡潔）。
- 良かった点：ユーザー回答の中で、{{IDEAL_POINTS}} と一致している点を1〜2個。
- 改善点：2〜3個、優先度順。各項目は以下の形：
  - 観察（ユーザーはこう書いている/書いていない）
  - 判定（{{IDEAL_POINTS}} に一致/不足/ズレ）
  - 次の一歩（次回、どの要点をどう具体化して書けば良いか）

- **模範解答を丸ごと提示しない**（方向性と不足要点の指摘に留める）。

"aspects": {

"data_flow": 0〜100の整数,
"error_handling": 0〜100の整数
}
}

## 評価基準（B以上を厳格化）

- A (90-100): {{IDEAL_POINTS}} をほぼ網羅し、要点に沿って具体的に説明できている（要点外の断定がほぼない）
- B (70-89): 主要要点を押さえ、概ね具体的に説明できている（不足や曖昧さは一部）
- C (50-69): 基本要点には触れるが、重要点の欠落や曖昧さ、要点外の主張が目立つ
- D (0-49): 要点が捉えられていない、一般論中心、または要点外の断定が多い
